Resumen Ejecutivo

Proyecto de inventario visual automatizado que utiliza modelos de visión-lenguaje (VLM) locales, ejecutados vía Ollama, para analizar fotografías de estanterías de supermercado y extraer una lista estructurada de productos con cantidades y precios.

Se iteró a través de 4 versiones del pipeline, logrando una mejora progresiva desde 0% de precisión global (v1) hasta 61% (v4), con una reducción del 79% en el sobre-conteo de productos.

Hallazgos Técnicos

3.1 Bug de Thinking Mode en Qwen3-VL

El modelo qwen3-vl:8b genera tokens <think>...</think> internos que la API OpenAI-compatible de Ollama filtra automáticamente. Cuando el modelo dedica toda su respuesta al "pensamiento", el contenido visible queda vacío.

Comportamiento: Prompts simples funcionan intermitentemente. Prompts que piden JSON estructurado producen respuestas vacías de forma consistente. El flag /no_think es ignorado en prompts complejos.

Solución: Se descartó qwen3-vl:8b y se adoptó qwen2.5vl:7b, que no tiene thinking mode y produce respuestas estables. Se migró a la API nativa de Ollama.

3.2 Truncamiento de JSON

Con num_predict=2000 tokens, las respuestas JSON se truncan frecuentemente cuando la estantería tiene muchos productos. En v4, el 63% de las respuestas (41/65) sufrieron truncamiento.

Mitigación: Parser con intentos de cierre automático (]}, "]}), fallback con regex para extraer productos del JSON parcial, y prompt con formato compacto ("n", "c", "e", "pr").

3.3 Sobre-conteo Sistemático

El modelo tiende a sobre-estimar cantidades, especialmente en estanterías con muchos productos repetidos (ej: 30 botellas de vodka similares).

Datos v1: 210 de 666 productos detectados tenían la cantidad máxima (15), indicando inflación masiva. En casos exitosos (prec. ≥ 80%), la cantidad promedio era 1-5. En casos fallidos (prec. < 30%), todos los productos estaban al tope.

3.4 Inconsistencia Inter-llamadas

Al dividir una imagen en 3 zonas (v3), el modelo nombra el mismo producto de forma diferente en cada zona (ej: "Baileys Original 700ml" vs "Baileys Irish Cream" vs "Licor Baileys"), generando duplicados que la deduplicación no resuelve.

Conclusión: Con modelos de 7B parámetros, la estrategia single-pass es superior a multi-pass.

Estrategia: 2 pasos (describir → JSON). Cap cantidad a 15. repeat_penalty=1.5.

Problema: Sobre-conteo masivo. El modelo repetía el mismo producto decenas de veces en el JSON (ej: "Licor 35 x3" repetido 47 veces).

Prec. global: 0.0% Sobre-conteo: +2,894

v2 — Few-shot + Cap Reducido

Cambios: Single-pass, few-shot example, cap a 8, post-corrección estadística, system prompt con rangos esperados.

Mejora clave: El few-shot example y el cap a 8 redujeron drásticamente el sobre-conteo.

Prec. global: 28.5% Prom/imagen: 39.6%

v3 — Split por Zonas Descartado

Hipótesis: Dividir imagen en 3 franjas horizontales y analizar cada una por separado.

Resultado: Empeoró respecto a v2. La inconsistencia inter-llamadas creó más duplicados de los que eliminó.

Prec. global: 13.3% Prom/imagen: 31.1% (↓8.5pp)

v4 — Prompt Refinado + Postprocess Dinámico Mejor

Cambios sobre v2: Ejemplo más realista (5 productos, 17 unidades), mensaje de "error común" explícito, temperatura 0.1, deduplicación fuzzy, post-corrección dinámica basada en productos únicos.

Mejora clave: Indicar explícitamente que "listar 50+ productos es INCORRECTO" redujo significativamente el sobre-conteo. Solo 4/65 imágenes necesitaron corrección estadística.

Lecciones Aprendidas

1

Los VLM de 7-8B tienen un techo para conteo preciso

Pueden identificar tipos de productos y leer etiquetas de precios, pero el conteo exacto de unidades individuales en estanterías densas sigue siendo un desafío.

2

El few-shot prompting es la mejora más efectiva

Mostrar un ejemplo concreto de respuesta correcta tuvo más impacto que cualquier técnica de post-procesamiento.

3

Indicar explícitamente los errores comunes ayuda

La línea "ERROR COMUN: listar 50+ productos es INCORRECTO" en el prompt de v4 redujo significativamente el sobre-conteo.

4

Multi-pass no funciona bien con modelos pequeños

La inconsistencia en cómo el modelo nombra productos entre llamadas genera más duplicados que los que resuelve.

5

Temperatura baja (0.1) mejora la consistencia del JSON

Reduce la variabilidad en la estructura de respuesta y los loops repetitivos.

6

El post-procesamiento estadístico es un safety net, no una solución

Solo 4 de 65 imágenes necesitaron corrección en v4, indicando que el prompt engineering fue más efectivo.

7

La API nativa de Ollama es más confiable

La capa OpenAI-compatible filtra contenido de modelos con comportamientos especiales como el thinking mode de Qwen3.

Decisiones de Diseño

Separación Core / Webapp: Las funciones de análisis (analyze_shelf, export_to_excel, etc.) se extrajeron a core.py como módulo independiente. Esto permite que tanto el CLI (analisis_estanteria.py) como la webapp (webapp.py) reutilicen exactamente la misma lógica sin duplicación.

Threading vs Celery: Se optó por threading.Thread para el procesamiento en background. Flask con un solo worker puede manejar las peticiones de estado mientras el hilo de análisis trabaja. Para un entorno productivo con múltiples usuarios simultáneos se necesitaría Celery + Redis, pero para este caso de uso es innecesario.

Puerto 5050: macOS reserva el puerto 5000 para AirPlay Receiver (servicio AirTunes). Se mapeó 5050:5000 en docker-compose para evitar el conflicto. Dentro del contenedor Flask sigue corriendo en el 5000 estándar.

Para usar la app: Ejecutar docker compose up -d y abrir http://localhost:5050 en el navegador.

Pipeline Híbrido SAM 3 + VLM

Se implementó un pipeline híbrido que combina Meta SAM 3 (Segment Anything Model 3, noviembre 2025) para detección y segmentación de productos con qwen2.5vl:7b para identificación de nombre y precio. A diferencia de las versiones v1–v4 donde el VLM hacía todo (detectar, contar, identificar), aquí cada modelo hace lo que mejor sabe hacer.

Arquitectura del Pipeline

Imagen (RGB)
    ↓
SAM 3 + text prompt: "product"
    ↓
    ├── mask_1 (score: 0.92, bbox)
    ├── mask_2 (score: 0.88, bbox)
    └── ...mask_N
    ↓  (filtrar: score ≥ 0.1, area ≥ 500px)
crop_mask_region() → PIL Image por mask
    ↓  (ThreadPoolExecutor, max_workers)
VLM (qwen2.5vl:7b) por cada crop:
    → {"name": "Vodka Absolut", "price": 15.99}
    ↓
deduplicate_products() → lista final → Excel

¿Por qué SAM 3 y no SAM 2.1?

SAM 3 (noviembre 2025) introduce text prompts para detección open-vocabulary: se le indica "product" y retorna masks solo de productos, no de estantes, separadores ni fondo. SAM 2.1 usaba segmentación ciega (AutomaticMaskGenerator) que generaba masks de todo lo visible, requiriendo filtrado agresivo posterior.

Hallazgos Técnicos de la Integración

SAM 3 no soporta CPU nativamente: El código de Meta tiene 6 hardcodes de device="cuda" y llamadas a pin_memory() que fallan sin GPU. Se requieren parches (incluidos en Dockerfile.sam) para ejecutar en CPU.

Dtype mismatch en CPU: SAM 3 carga pesos en bfloat16 y espera torch.autocast("cuda"). En CPU se necesita torch.autocast("cpu", dtype=torch.bfloat16) para evitar errores de tipos.

Modelo gated: El checkpoint de SAM 3 en HuggingFace (facebook/sam3) requiere aceptar la licencia y un token HF_TOKEN. La aprobación puede no ser instantánea.

Dependencias no declaradas: SAM 3 requiere einops, pycocotools y psutil que no están en su setup.py. Se instalan por separado en el Dockerfile.

Threshold 0.1 vs 0.5: Con confidence=0.5 el prompt "product on supermarket shelf" detecta 0 masks. Con confidence=0.1 y prompt "product" detecta ~100-200 masks. Los scores en CPU son más bajos que en GPU.

Ejecución

# Prerrequisito: aceptar licencia en https://huggingface.co/facebook/sam3
export HF_TOKEN=hf_xxxxx

# Construir y levantar
docker compose -f docker-compose.sam.cpu.yml up -d --build

# Analizar una imagen
docker exec -e HF_TOKEN=$HF_TOKEN entorno-sam \
  python /app/app/analisis_estanteria_hybrid.py \
  --image /archive/images/0058.jpg --sam-device cpu

# Analizar muestra del dataset
docker exec -e HF_TOKEN=$HF_TOKEN entorno-sam \
  python /app/app/analisis_estanteria_hybrid.py \
  --sample 1 --max-workers 1 --text-prompt "product" --confidence 0.1

Mejoras Futuras

Optimizar parámetros SAM 3

Ajustar --confidence y --max-masks usando el ground truth completo (15,064 anotaciones) para reducir el sobre-conteo actual (+100%).

SAM 3.1 Object Multiplex

Versión de marzo 2026 con 7x más velocidad para multi-objeto. Ideal para estanterías con 20+ productos.

Fine-tuning de SAM 3

Entrenar con las 15,064 anotaciones del dataset para mejor segmentación específica de productos en estanterías.

GPU NVIDIA

SAM 3 en GPU reduciría el tiempo de ~200s a ~20-30s por imagen y mejoraría los scores de confianza.

VLM más grande

Modelos de 13B+ parámetros para mejor identificación de productos y lectura de etiquetas de precio.

Ensemble de prompts

Combinar resultados de múltiples text prompts ("product", "bottle", "item on shelf") para mayor cobertura.
