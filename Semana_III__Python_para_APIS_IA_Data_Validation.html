Introducción: Arquitectura de Validación de Datos

La transición de lenguajes orientados exclusivamente al análisis exploratorio de datos (EDA) y la modelización matemática teórica hacia entornos de producción y automatización en Python requiere un cambio de paradigma estructural. En la estadística clásica, el análisis interactivo tolera un alto grado de ambigüedad estructural; los tipos de datos pueden inferirse dinámicamente en tiempo real, y las anomalías o violaciones de supuestos se manejan de manera visual o analítica a posteriori mediante la inspección de residuales. Sin embargo, en la ingeniería de datos moderna y el despliegue de modelos estadísticos de aprendizaje automático, la ingestión de datos debe someterse a un paradigma estrictamente confirmatorio desde el primer momento de ejecución.

La falta de rigor en la definición inicial de la estructura de los datos de entrada conduce inevitablemente a errores de ejecución silentes, sesgos de ingestión por coerción automática de variables y, en última instancia, a inferencias causales espurias. El presente documento establece un marco metodológico exhaustivo para aplicar principios de ingeniería de software —específicamente el tipado estático avanzado de Python 3.11 y la validación de esquemas mediante la biblioteca Pydantic V2— traduciendo la nomenclatura técnica de los sistemas informáticos al lenguaje estandarizado del diseño experimental, la biometría y el control de calidad estadístico.

Módulo 1: Tipos Complejos y Anotaciones de Datos Estadísticos

1.1 Fundamentos Teóricos

1.1.1 El Libro de Códigos Computacional

Declarar y definir tipos estáticos en un entorno de programación como Python constituye el equivalente algorítmico a establecer el "libro de códigos" (codebook) exhaustivo o el diccionario de variables de una encuesta estructurada antes de iniciar el trabajo de campo empírico. Así como un investigador restringe una variable de escala de Likert para que el encuestador solo acepte números enteros discretos en un intervalo cerrado, el tipado estático asegura que una variable categórica codificada no sea tratada incidentalmente como una variable continua por el procesador, previniendo operaciones matemáticas ilógicas como calcular la media aritmética de un vector de códigos postales o identificadores de paciente.

¿Qué es una Escala Likert?

La escala de Likert es una de las herramientas de medición más utilizadas en la investigación cuantitativa (psicometría, encuestas de opinión). Su propósito central es medir el nivel de acuerdo o desacuerdo de una persona frente a una afirmación específica (ítem).

Clásicamente y a nivel estadístico asume una naturaleza puramente ordinal estructurada simétricamente y por lo general comprende cinco a siete opciones discretas, por ejemplo:

  1. Totalmente en desacuerdo
  2. En desacuerdo
  3. Ni de acuerdo ni en desacuerdo (Neutral)
  4. De acuerdo
  5. Totalmente de acuerdo

* Computacionalmente esto implica que un validador debería restringir estos valores a un arreglo entero finito y definido como [1, 2, 3, 4, 5].

Definición 1.1 (Tipo Estático): Un tipo estático es una anotación declarativa que especifica el dominio de valores permitidos para una variable antes de la ejecución del programa. En términos estadísticos, el tipo define el espacio muestral \( \Omega \) asociado a una variable aleatoria \( X \), donde \( X: \Omega \rightarrow D \), siendo \( D \) el dominio de valores válidos.

1.1.2 Notación Matemática para Tipos de Datos

En Python 3.11, el sistema de tipos ha evolucionado sustancialmente para permitir una especificidad que rivaliza con las definiciones algebraicas rigurosas. Los tipos complejos fundamentales se representan mediante la siguiente notación:

Tipo Python Notación Matemática Interpretación Estadística
List[T] \( X = (X_1, X_2, \dots, X_n) \in T^n \) Vector univariado de observaciones estrictamente homogéneas. Análogo computacional de un vector columna en una matriz de diseño.
Union[A, B] \( X \in A \cup B \) Variable aleatoria que puede pertenecer válidamente al espacio muestral A o al espacio muestral B.
Optional[T] \( X \in T \cup \{\emptyset\} \) Variable que puede contener el valor empírico esperado o representar explícitamente un dato perdido (None/NULL).
npt.NDArray[np.float64] \( X \in \mathbb{R}^{m \times n} \) Matriz de diseño con elementos de precisión doble (64 bits), optimizada para cálculos de álgebra lineal.
Ecuación 1.1: Definición formal del tipo Optional como unión disjunta

$$ \text{Optional}[T] \equiv T \biguplus \{\text{None}\} \equiv \text{Union}[T, \text{None}] $$
Inmersión Teórica: Interpretación de la Unión Disjunta

La notación \( \biguplus \) (unión disjunta) es fundamental en la teoría de tipos computacional y en la teoría formal de conjuntos. A diferencia de una unión clásica \( \cup \), la unión disjunta garantiza estructuralmente que los elementos del conjunto \( T \) (por ejemplo, números reales \( \mathbb{R} \)) jamás se solapan o confunden con el estado de ausencia representado por el elemento unitario especial None (nulo).

Interpretación Semántica y Estadística:

  • Cualquier observación modelada como Optional[T] obliga al analista y al motor de evaluación estadístico a procesar explícita y obligatoriamente la bifurcación mutuamente excluyente del dato: o bien es una observación válida perteneciente a la distribución empírica \( T \), o es certera, deliberada y demostrablemente un dato faltante estructural (el conjunto {None}).
  • Esta separación categórica impide formalmente que un valor nulo sea silenciosamente casteado o "coercido" a cero (\( 0 \)) o a NaN durante las operaciones de álgebra matricial subyacentes. Como corolario, obliga por diseño a invocar rutinas explícitas de imputación estocástica, eliminación por casewise deletion, o estimaciones robustas bajo asunciones de MAR (Missing At Random), blindando de antemano la muestra frente a sesgos de ingestión.

1.1.3 Tipos Genéricos Parametrizados

Con la introducción formal de TypeVarTuple mediante el PEP 646 y el paquete typing_extensions, es matemáticamente posible especificar la heterogeneidad columnar y la dimensionalidad exacta de las matrices. Esto garantiza que una matriz de diseño X suministrada a un algoritmo tenga exactamente las dimensiones requeridas y los tipos escalares correctos antes de intentar calcular los estimadores por Mínimos Cuadrados Ordinarios o la inversión de matrices.

Ecuación 1.2: Especificación dimensional de matriz de diseño

$$ X \in \mathbb{R}^{n \times p} \quad \text{donde } n = \text{número de observaciones, } p = \text{número de covariables} $$

# 1. Definición del tipo para matriz de covariables continuas
import numpy.typing as npt

# Anotación de arreglos multidimensionales con precisión especificada
MatrizDisenoCovariables = npt.NDArray[np.float64]

# 2. Definición del modelo de observación tipado
class EspecificacionCoche(BaseModel):
    """
    Modelo de validación para una observación individual del dataset mtcars.
    Cada campo representa una variable del diseño experimental.
    """
    # Identificador nominal del sujeto (puede estar perdido)
    modelo_vehiculo: Optional[str] = Field(
        default=None, 
        description="Identificador nominal del sujeto experimental"
    )
    
    # Caballos de fuerza: permite polimorfismo int/float
    caballos_fuerza_hp: int | float = Field(
        ..., 
        description="Medición de potencia bruta del motor",
        gt=0  # Restricción: debe ser positivo
    )
    
    # Vector de mediciones repetidas del cuarto de milla
    qsec_mediciones_repetidas: List[float] = Field(
        default_factory=list,
        description="Tiempos en cuarto de milla (medidas repetidas)"
    )
    
    # Variable indicadora con restricciones embebidas
    transmision_am_dummy: Annotated[int, Field(
        ge=0, le=1, 
        description="Variable dummy: 0 = Automático, 1 = Manual"
    )]

# 3. Función con firma tipada para ajuste de modelo
def ajustar_modelo_lineal(
    X: MatrizDisenoCovariables,
    y: npt.NDArray[np.float64],
    datos_completos: pd.DataFrame
) -> dict:
    """
    Función que exige matriz de diseño estrictamente numérica flotante.
    Calcula estimadores básicos de regresión lineal.
    """
    n, p = X.shape
    
    # Cálculo de estimadores por mínimos cuadrados ordinarios
# \( \hat{\beta} = (X'X)^{-1}X'y \)
    XtX_inv = np.linalg.inv(X.T @ X)
    beta_hat = XtX_inv @ (X.T @ y)
    
    # Residuales y estimación de varianza
    y_pred = X @ beta_hat
    residuales = y - y_pred
    sigma2_hat = np.sum(residuales**2) / (n - p)
    
    return {
        'coeficientes': beta_hat,
        'varianza_residual': sigma2_hat,
        'n_observaciones': n,
        'p_parametros': p
    }

# 4. Instanciación y validación de una observación
coche_valido = EspecificacionCoche(
    modelo_vehiculo="Mazda RX4",
    caballos_fuerza_hp=110.0,
    qsec_mediciones_repetidas=[16.46, 16.50, 16.48],
    transmision_am_dummy=1
)

print(f"✅ Vehículo validado: {coche_valido.modelo_vehiculo}")
print(f"   HP: {coche_valido.caballos_fuerza_hp}")
print(f"   Transmisión: {'Manual' if coche_valido.transmision_am_dummy else 'Automático'}")

Interpretación de Resultados

La implementación de firmas de tipo transforma el código estadístico de una declaración de intenciones implícita a un contrato matemático explícito y auditable. Cuando MatrizDisenoCovariables se define explícitamente como npt.NDArray[np.float64], herramientas de análisis estático como mypy o pyright pueden predecir fallos de incompatibilidad dimensional o de tipo de datos sin necesidad de ejecutar simulaciones costosas. La función ajustar_modelo_lineal garantiza que los cálculos de álgebra lineal (como la inversión de matrices X'X) operen exclusivamente sobre datos de punto flotante de 64 bits, optimizando el rendimiento del motor BLAS/LAPACK subyacente.

Módulo 2: Validadores de Campo (@field_validator)

2.1 Fundamentos Teóricos

2.1.1 El Filtro Paramétrico de Criterios de Exclusión

Un validador de campo en la biblioteca Pydantic (invocado mediante el decorador @field_validator) opera bajo el mismo nivel de rigor y escrutinio metodológico que las pruebas lógicas de inclusión y exclusión protocolizadas en un ensayo clínico aleatorizado de fase III. Si un paciente reclutado —equivalente a un vector de características xᵢ— presenta un peso corporal negativo, o una medición de saturación de oxígeno en sangre superior al 100%, esta observación empírica viola indiscutiblemente los límites teóricos del dominio fisiológico.

Definición 2.1 (Validador de Campo): Un validador de campo es una función decorada que intercepta el valor asignado a un atributo antes o después de la coerción de tipos, aplicando reglas lógicas que verifican el cumplimiento de axiomas matemáticos o restricciones del dominio teórico. Matemáticamente, sea \( v: D \rightarrow D \cup \{\text{Error}\} \) la función validadora, donde D es el dominio de valores permitidos.

2.1.2 Modalidades de Validación

Existen dos filosofías operativas principales de validación configurables en el motor Pydantic V2:

Modalidad Descripción Equivalente Metodológico
Lax Mode (Por defecto) El algoritmo intenta coerción probabilística (ej. transforma "4" al entero 4) Investigador de campo que limpia errores tipográficos obvios antes de tabular
Strict Mode Exige que el tipo de dato sea exacto desde su origen, sin conversiones Inferencia no paramétrica que rechaza imponer distribuciones teóricas sobre datos que no encajan perfectamente

2.1.3 Temporalidad de la Validación

Los validadores pueden instruirse para operar temporalmente antes (mode='before') o después (mode='after') de la coerción de tipos:

Ecuación 2.1: Flujo de validación temporal

$$ \text{Entrada cruda} \xrightarrow{\text{mode='before'}} \text{Coerción de tipos} \xrightarrow{\text{mode='after'}} \text{Valor validado} $$
  • mode='before': Recibe el dato crudo, sin tipar. Ideal para estandarizar formatos de ingreso exóticos, extraer substrings mediante expresiones regulares, o corregir problemas de internacionalización.
  • mode='after': Opera sobre un tipo de dato ya seguro y coercido. Permite enfocarse exclusivamente en comprobar la regla matemática o delimitación del rango paramétrico.

2.1.4 Rendimiento del Motor de Validación

El ecosistema Pydantic V2 reescribió íntegramente su motor de validación base utilizando el lenguaje de programación Rust, lo que permite ejecutar estas evaluaciones lógicas complejas con un incremento masivo de rendimiento temporal, oscilando entre 5 a 50 veces mayor velocidad en comparación con las iteraciones anteriores basadas puramente en Python. Esta arquitectura es de importancia crítica cuando se requiere evaluar iterativamente millones de observaciones paramétricas generadas mediante técnicas como Bootstrapping, validación cruzada de K-pliegues, o simulaciones MCMC.

¿Qué es una simulación MCMC (Markov Chain Monte Carlo)?

Las simulaciones Markov Chain Monte Carlo (MCMC) son una clase de algoritmos computacionales estadísticos fundamentales en Inferencia Bayesiana avanzada.

Este método combina dos conceptos:

  • Cadenas de Markov: Sistemas donde evaluar el estado "siguiente" depende únicamente del estado "actual", ignorando toda la historia pasada.
  • Monte Carlo: Una técnica que emplea muestreos aleatorios repetidos masivamente para aproximar resultados u operar integrales complejas.

Aplicación Práctica: En lugar de intentar resolver analíticamente una ecuación irresoluble para encontrar la distribución de probabilidad a posteriori precisa (posterior distribution), el MCMC permite "explorar" el espacio numérico saltando estadísticamente de un valor a otro hasta mapear exhaustivamente la topología de la curva, requiriendo generar iterativamente millones de pequeños "pasos" u observaciones aleatorias (que la arquitectura de Pydantic descrita necesita validar rápidamente).

El conjunto de datos iris compilado por Edgar Anderson y popularizado por Ronald Fisher contiene mediciones biológicas precisas de taxones florales. Es un axioma biológico irrefutable que la longitud de un sépalo y un pétalo floral deben registrarse como valores continuos estrictamente positivos. Se implementará un validador que imponga esta restricción teórica fundamental, acoplado con un decorador para medir la eficiencia temporal del proceso de validación.

¿Qué es el Bootstrapping y por qué se aplica en este ejemplo? ▼

Bootstrapping es una técnica estadística de remuestreo introducida por Bradley Efron (1979) que consiste en generar múltiples muestras artificiales a partir de una muestra original, extrayendo observaciones con reemplazo. Cada muestra generada (llamada muestra bootstrap) tiene el mismo tamaño que la original, pero al extraer con reemplazo, algunas observaciones se repiten y otras quedan fuera.

¿Para qué sirve? Permite estimar la distribución muestral de un estadístico (media, varianza, intervalos de confianza, etc.) sin necesidad de asumir una distribución teórica subyacente ni de recolectar nuevas muestras del mundo real. En esencia, la muestra original actúa como un "mini-universo" del cual se extraen réplicas computacionales.

En el ejemplo que sigue, el bootstrapping se utiliza con un propósito doble:

  1. Simular un escenario realista de ingesta masiva: En producción, una API recibe cientos o miles de registros por segundo. Al generar múltiples réplicas bootstrap de las 150 observaciones originales de Iris, se simula un flujo masivo de datos que estresa el sistema de validación Pydantic de manera representativa.
  2. Evaluar la robustez del validador ante datos ruidosos: Al inyectar deliberadamente errores en algunas observaciones (valores negativos, fuera de rango, especies inválidas), se verifica que el pipeline de validación detecte y rechace consistentemente los datos corruptos sin importar cuántas veces aparezcan en el remuestreo.

Adicionalmente, el decorador @cronometrar_bootstrap que se implementa en el código mide el tiempo de ejecución de todo el proceso, lo cual es relevante porque en un escenario real de bootstrapping estadístico (miles de iteraciones de validación) la eficiencia computacional es un factor crítico.

Implementación Paso a Paso

Python 
import time
import functools
from typing import ClassVar, List
from pydantic import BaseModel, field_validator, ValidationError

# 1. Decorador para cronometrar rutinas de validación masiva
def cronometrar_bootstrap(func):
    """
    Decorador que mide el tiempo de ejecución de rutinas iterativas
    como validaciones masivas o remuestreos de Bootstrap.
    """
    @functools.wraps(func)
    def envoltorio_tiempo(*args, **kwargs):
        tiempo_inicio = time.perf_counter()
        resultado = func(*args, **kwargs)
        tiempo_fin = time.perf_counter()
        duracion = tiempo_fin - tiempo_inicio
        print(f"La rutina '{func.__name__}' ejecutó en {duracion:.5f} segundos.")
        return resultado
    return envoltorio_tiempo

# 2. Modelo de validación para observaciones del dataset iris
class ObservacionIris(BaseModel):
    """
    Modelo estadístico para validación de observaciones florales.
    Incluye validadores para garantizar axiomas biológicos.
    """
    # Variable dependiente (factor categórico)
    especie_taxonomica: str
    
    # Vector de variables predictoras continuas
    sepal_length: float
    sepal_width: float
    petal_length: float
    petal_width: float
    
    # Constantes de clase: niveles permitidos del factor
    ESPECIES_PERMITIDAS: ClassVar[list] = ["setosa", "versicolor", "virginica"]
    
    @field_validator('especie_taxonomica')
    @classmethod
    def validar_factor_categorico(cls, valor: str) -> str:
        """
        Garantiza que el factor categórico solo contenga niveles
        poblacionales conocidos. Previene 'niveles fantasma' que
        destruirían los grados de libertad en análisis posteriores.
        """
        valor_limpio = valor.strip().lower()
        if valor_limpio not in cls.ESPECIES_PERMITIDAS:
            raise ValueError(
                f"Anomalía Taxonómica. Especie no reconocida: '{valor}'. "
                f"El hiperespacio poblacional permite: {cls.ESPECIES_PERMITIDAS}"
            )
        return valor_limpio
    
    @field_validator('sepal_length', 'sepal_width', 'petal_length', 'petal_width', mode='after')
    @classmethod
    def auditar_axiomas_biologicos(cls, valor: float) -> float:
        """
        Comprueba axiomas del dominio del proceso generador de datos:
        las medidas morfológicas deben ser estrictamente positivas.
        """
        # Restricción teórica: valores positivos
        if valor <= 0:
            raise ValueError(
                "Violación Axiomática: Las mediciones biológicas deben "
                "residir en la región topológica \(0, +\infty\)."
            )

        # Heurística de control de calidad: detectar errores de escala
        if valor > 50.0:
            raise ValueError(
                f"Medición atípica sospechosa ({valor} cm); posible error "
                "de transcripción de milímetros a centímetros."
            )
        return valor

# 3. Simulación de ingesta muestral masiva con manejo de errores
@cronometrar_bootstrap
def simular_ingesta_bootstrap(lista_datos: list) -> dict:
    """
    Procesa una lista de observaciones simulando un pipeline de ingesta
    con validación automatizada y registro de exclusiones.
    """
    observaciones_validas = []
    errores_detectados = []
    
    for i, dato in enumerate(lista_datos):
        try:
            medicion = ObservacionIris(**dato)
            observaciones_validas.append(medicion)
        except ValidationError as error:
            errores_detectados.append({
                'indice': i,
                'dato_original': dato,
                'error': str(error)
            })
            print(f"\n⚠️ [Rechazo #{len(errores_detectados)}] Observación índice {i}")
    print(f"   Error: {error.errors()[0]['msg'][:80]}...")
    
    return {
        'validas': observaciones_validas,
        'rechazadas': errores_detectados,
        'tasa_exclusion': len(errores_detectados) / len(lista_datos) * 100
    }

# 4. Ejecución de la simulación
resultado = simular_ingesta_bootstrap(datos_iris_simulados)
print(f"\n📊 Resumen de Validación:")
print(f"   Observaciones aceptadas: {len(resultado['validas'])}")
print(f"   Observaciones rechazadas: {len(resultado['rechazadas'])}")
print(f"   Tasa de exclusión: {resultado['tasa_exclusion']:.1f}%")

¿Qué son MCAR, MAR y MNAR? La taxonomía de Rubin para datos faltantes ▼

En 1976, el estadístico Donald B. Rubin propuso una clasificación formal de los mecanismos que generan datos faltantes (missing data) en un conjunto de datos. Esta taxonomía de Rubin es fundamental porque el tipo de mecanismo determina qué técnicas de análisis son válidas y si las conclusiones estadísticas obtenidas con los datos disponibles son generalizables a la población original.

Formalmente, si \(\mathbf{Y}\) es la matriz completa de datos y \(\mathbf{R}\) es la matriz indicadora de observación (donde \(R_{ij} = 1\) si el dato \(Y_{ij}\) fue observado y \(R_{ij} = 0\) si está faltante), la clasificación se basa en la distribución condicional \(P(\mathbf{R} \mid \mathbf{Y})\):

MCAR — Missing Completely At Random (Faltante Completamente al Azar)
$$ P(\mathbf{R} \mid \mathbf{Y}_{\text{obs}}, \mathbf{Y}_{\text{mis}}) = P(\mathbf{R}) $$

La probabilidad de que un dato falte no depende de ningún valor, ni observado ni faltante. La ausencia es puramente aleatoria, como si se lanzara una moneda para decidir qué datos eliminar.

Ejemplo: Un sensor de temperatura falla por una descarga eléctrica aleatoria que no tiene relación con la temperatura que estaba midiendo. Los datos perdidos son una submuestra aleatoria de los datos completos.

Implicación: Es el escenario más benigno. Los análisis con datos completos (sin los faltantes) producen estimadores insesgados, aunque con menor potencia estadística por la reducción del tamaño muestral.

MAR — Missing At Random (Faltante al Azar)
$$ P(\mathbf{R} \mid \mathbf{Y}_{\text{obs}}, \mathbf{Y}_{\text{mis}}) = P(\mathbf{R} \mid \mathbf{Y}_{\text{obs}}) $$

La probabilidad de que un dato falte depende de otros valores observados, pero no del valor faltante en sí. El nombre es algo engañoso: no es "completamente al azar", sino "al azar condicional" a lo observado.

Ejemplo: En una encuesta de ingresos, las personas jóvenes (edad observable) tienden a no reportar su salario, pero dentro de cada grupo de edad, la probabilidad de no reportar no depende del salario real. La ausencia se explica por la edad, no por el ingreso.

Implicación: Eliminando simplemente los registros incompletos se introduce sesgo. Sin embargo, técnicas como imputación múltiple o modelos de máxima verosimilitud que condicionen en las variables observadas pueden producir estimaciones válidas.

MNAR — Missing Not At Random (Faltante No al Azar)
$$ P(\mathbf{R} \mid \mathbf{Y}_{\text{obs}}, \mathbf{Y}_{\text{mis}}) \neq P(\mathbf{R} \mid \mathbf{Y}_{\text{obs}}) $$

La probabilidad de que un dato falte depende del propio valor faltante, incluso después de condicionar por las variables observadas. Este es el mecanismo más problemático.

Ejemplo: En un estudio clínico, los pacientes con efectos secundarios más graves abandonan el ensayo y dejan de reportar sus síntomas. Los datos faltantes son precisamente los más extremos, y su ausencia distorsiona la distribución observada.

Implicación: No existe una solución puramente estadística sin supuestos adicionales. Se requieren modelos de selección (selection models) o modelos de mezcla de patrones (pattern-mixture models) que incorporen hipótesis explícitas sobre el mecanismo de ausencia. Sin este tratamiento, cualquier análisis producirá estimaciones sesgadas.

¿Por qué importa esto en nuestro pipeline de validación?

Cuando Pydantic rechaza datos, está generando "datos faltantes" de forma programática. Si las reglas de validación son puramente estructurales (tipo incorrecto, campo vacío), los rechazos tienden a ser MCAR. Pero si los rechazos se concentran en valores extremos reales (ej.: rechazar edades > 120 elimina centenarios legítimos), el mecanismo se convierte en MNAR y el dataset resultante queda sesgado. Por eso la bitácora de auditoría y el análisis de patrones de exclusión son indispensables: permiten diagnosticar qué tipo de mecanismo de Rubin está operando y tomar decisiones informadas antes de proceder con el análisis estadístico.

Rol del decorador @cronometrar_bootstrap: El tiempo de ejecución registrado permite evaluar la viabilidad computacional de esta estrategia de validación en escenarios de producción. Si la validación de 15 observaciones toma \(t\) segundos, un bootstrap de 10,000 réplicas requeriría aproximadamente \(667 \times t\) segundos. Cuando este valor supera umbrales operativos, es señal de que se deben considerar optimizaciones como validación por lotes, procesamiento asíncrono o paralelización.

Valor del diccionario de retorno: La función simular_ingesta_bootstrap devuelve un diccionario con tres claves: validas (lista de instancias ObservacionIris certificadas), rechazadas (lista de diccionarios con el índice, dato original y mensaje de error) y tasa_exclusion (porcentaje). Esta estructura constituye una bitácora de auditoría mínima que permite reconstruir exactamente qué se descartó y por qué, requisito fundamental para reportes de trazabilidad metodológica (CONSORT/STROBE).

Ejercicio 1: Diferencia entre mode='before' y mode='after'

Problema: Explicar la diferencia axiomática y pragmática entre parametrizar mode='before' y mode='after' en un @field_validator cuando se enfrenta a la limpieza de distribuciones empíricas ruidosas que pueden contener valores como "1,234.56" (coma como separador de miles), "-25" (edad negativa), o "NA" (dato faltante no estandarizado).

Solución Parcial:

La configuración mode='before' recibe el punto de datos crudo y potencialmente no tipado antes de cualquier coerción. Es ideal para:

  • Estandarizar formatos de ingreso exóticos (convertir "1,234.56" a "1234.56")
  • Manejar representaciones no estandarizadas de datos faltantes (convertir "NA", "N/A", "-" a None)
  • Corregir problemas de internacionalización (comas vs. puntos decimales)

Por el contrario, mode='after' opera sobre un dato ya coercido y tipado, permitiendo enfocarse exclusivamente en validar reglas matemáticas (rango, dominio) sin preocuparse por fallos de conversión.

Completar: Implementar un validador mode='before' que maneje el caso especial donde la edad puede venir como "desconocida", "N/A", o como entero negativo, convirtiendo todos estos casos a None.

2.5 Perspectiva Complementaria: Dead Letter Queue con Celery

¿Qué es Celery y su DLQ?

A lo largo de este módulo hemos implementado un mecanismo de Dead Letter Queue (DLQ) manual e in-process: cuando Pydantic rechaza un dato, lo capturamos en un diccionario Python y lo almacenamos en una lista local dentro del mismo proceso. Este enfoque es didácticamente claro y funcional para scripts de análisis, pero presenta limitaciones importantes en entornos de producción distribuidos.

Celery es un sistema de colas de tareas distribuidas para Python que permite ejecutar trabajos de forma asíncrona en procesos separados (workers), orquestados por un message broker como RabbitMQ o Redis. En el ecosistema de Celery, una Dead Letter Queue es una cola especial donde se depositan automáticamente las tareas que fallan después de agotar sus reintentos, sin intervención manual del programador.

Arquitectura de RabbitMQ: Exchanges, Bindings y Queues

Para comprender cómo funciona una DLQ nativa, es necesario entender los tres componentes fundamentales de RabbitMQ (Chong, 2025):

Exchange

Actúa como un enrutador de mensajes. El productor nunca envía directamente a una cola; publica en un exchange que decide a qué cola(s) encaminar el mensaje según reglas de enrutamiento (routing keys). Tipos comunes: direct, topic, fanout.

Binding

Es la regla de enlace entre un exchange y una cola. Define qué routing key debe coincidir para que los mensajes fluyan del exchange a la cola. Sin binding, los mensajes publicados en un exchange se descartan.

Queue

Es el buffer de mensajes donde se almacenan hasta que un consumidor (worker) los procesa. Soporta argumentos especiales como x-dead-letter-exchange para configurar DLQ nativa.

El flujo completo es: Productor → Exchange → (Binding + Routing Key) → Queue → Consumidor. Cuando un mensaje falla o expira en la cola principal, RabbitMQ lo redirige automáticamente al dead letter exchange configurado.

DLQ Nativa de RabbitMQ: Argumentos de Cola

RabbitMQ soporta DLQ de forma nativa mediante argumentos de cola (queue arguments) que se definen al momento de declarar la cola. Los tres argumentos clave son:

x-dead-letter-exchange

Define el exchange al que se redirigen los mensajes rechazados o expirados. Es el "destino de fallback".

x-dead-letter-routing-key

Especifica la routing key que se usa al publicar en el dead letter exchange (permite enrutar a una cola DLQ específica).

x-message-ttl

Tiempo de vida (en milisegundos) de los mensajes en la cola. Al expirar, se envían al dead letter exchange automáticamente.

Comparativa: DLQ Manual (Pydantic) vs. DLQ de Celery

Aspecto DLQ Manual (este material) DLQ con Celery + RabbitMQ
Arquitectura In-process: todo ocurre en un solo proceso Python (lista en memoria). Distribuida: broker externo (RabbitMQ/Redis) + workers independientes.
Persistencia Volátil: si el proceso muere, los datos rechazados se pierden. Persistente: el broker almacena los mensajes en disco hasta que se consumen.
Reintentos automáticos Manual: el programador debe implementar la lógica de reintento. Nativo: autoretry_for, max_retries, backoff exponencial.
Mecanismo DLQ Lógico: lista Python en memoria controlada por try/except. Nativo del broker: x-dead-letter-exchange + x-dead-letter-routing-key a nivel de cola RabbitMQ.
Escalabilidad Limitada al proceso y la RAM disponible. Horizontal: se agregan workers sin modificar código.
Complejidad de setup Baja: solo Python + Pydantic. Media-Alta: requiere instalar y configurar broker, workers, backend de resultados.
Monitoreo Manual: print() o logging básico. Integrado: Flower (dashboard web), métricas Prometheus, alertas.

Optimización de Workers Celery + RabbitMQ

Chong (2025) identifica tres configuraciones críticas que transforman el comportamiento de Celery en producción. Sin ellas, los workers pueden perder mensajes silenciosamente o distribuir la carga de forma desigual:

worker_prefetch_multiplier = 1

Por defecto, Celery pre-descarga (prefetch) 4 mensajes por worker. Si las tareas tienen duración variable, esto causa desbalanceo: un worker acapara tareas largas mientras otros están ociosos. Con prefetch_multiplier=1, cada worker solo toma una tarea a la vez, distribuyendo la carga equitativamente.

task_acks_late = True

Por defecto, Celery envía el ACK (acknowledgement) al broker apenas recibe el mensaje, antes de procesarlo. Si el worker muere durante la ejecución, el mensaje se pierde. Con task_acks_late=True, el ACK se envía después de completar la tarea, garantizando que los mensajes no procesados permanezcan en la cola.

task_reject_on_worker_lost = True

Complementa a task_acks_late. Si un worker se pierde (crash, OOM kill, desconexión), el mensaje se rechaza activamente (NACK) y RabbitMQ lo reencola o lo redirige a la DLQ según la configuración de x-dead-letter-exchange. Sin esta opción, los mensajes quedarían en estado "unacked" indefinidamente.

Regla práctica: Estas tres opciones deben activarse siempre en conjunto. task_acks_late sin task_reject_on_worker_lost puede causar mensajes huérfanos; prefetch_multiplier=1 es esencial cuando las tareas de validación tienen tiempos de ejecución variables (como ocurre con datos que requieren múltiples reintentos).

Módulo 3: Modelos Anidados y Topología Jerárquica de Datos

3.1 Fundamentos Teóricos

3.1.1 El Diseño de Parcelas Divididas en Memoria Computacional

Un modelo de datos anidado (Nested Model) en ingeniería de software orientada a objetos es topológica y estructuralmente isomorfo a un diseño experimental de parcelas divididas (split-plot design) utilizado en investigaciones agrícolas, o a los Modelos Lineales Jerárquicos de Efectos Mixtos (HLM / Mixed-Effects Models) formulados en la psicometría clásica.

¿Qué significa "topológica y estructuralmente isomorfo"?

Esta expresión combina tres conceptos que, juntos, dicen algo bastante simple. Vamos por partes:

1. "Isomorfo" — La idea central

En matemáticas, dos cosas son isomorfas cuando tienen la misma forma aunque parezcan diferentes por fuera. Imagine dos organigramas de empresas distintas: una es un hospital y otra es una universidad. Los nombres y las funciones son diferentes, pero si ambas tienen un director general arriba, tres subdirectores debajo, y cada subdirector tiene cinco jefes de departamento, entonces la forma de ambos organigramas es idéntica. Eso es un isomorfismo: una correspondencia uno-a-uno que preserva las relaciones.

2. "Topológica" — La forma de las conexiones

La topología estudia cómo están conectadas las cosas entre sí, sin importar distancias ni tamaños. Piense en un mapa del metro: no refleja las distancias reales entre estaciones, pero sí muestra correctamente cuáles estaciones están conectadas y en qué orden. Cuando decimos que algo es "topológicamente" igual, nos referimos a que el patrón de conexiones —qué contiene a qué, qué depende de qué— es el mismo.

3. "Estructuralmente" — La organización interna

Este término refuerza que no solo las conexiones son iguales, sino también la organización jerárquica: cuántos niveles hay, cómo se agrupan los elementos, y qué reglas rigen cada nivel. Es la diferencia entre decir "están conectados igual" y "además están organizados igual".

Entonces, ¿qué dice el párrafo?

Dice que un modelo de datos anidado en Python (por ejemplo, una clase SesionDinamometro que contiene una lista de EnsayoVehiculo) tiene exactamente la misma forma y organización que un diseño experimental de parcelas divididas en agricultura (una parcela grande dividida en sub-parcelas). Aunque uno vive en la memoria de un computador y el otro en un campo de cultivo, la manera en que se organizan los niveles, las agrupaciones y las dependencias es idéntica.

Ambas estructuras son isomorfas: misma forma, mismas conexiones, misma jerarquía — diferente contexto.

Definición 3.1 (Modelo Anidado): Un modelo anidado es una estructura de datos donde una clase contenedora (nivel superior) encapsula colecciones de clases dependientes (nivel inferior), estableciendo una relación de composición que refleja la estructura jerárquica del diseño experimental. Matemáticamente, si $B$ es el conjunto de bloques y $U$ es el conjunto de unidades observacionales, entonces existe una función $f: U \rightarrow B$ que asigna cada unidad a su bloque correspondiente.

3.1.2 Notación para Estructuras Jerárquicas

En el contexto de diseños experimentales jerárquicos, la notación matemática estándar expresa la estructura de anidamiento mediante subíndices:

Ecuación 3.1: Modelo jerárquico básico

$$ \mathbf{Y}_{ijk} = \mu + \alpha_i + \beta_{j(i)} + \varepsilon_{k(ij)} $$

donde:
• \( \alpha_i \) = efecto del nivel superior (bloque i)
• \( \beta_{j(i)} \) = efecto del nivel j anidado en i
• \( \varepsilon_{k(ij)} \) = error de la observación k anidada en ij

3.1.4 Preservación de la Covarianza Intra-Clase

Históricamente, el reflejo automático del analista consistía en aplanar (flatten) prematuramente esta rica estructura jerárquica tridimensional en una única tabla rectangular, perdiendo en el proceso la topología de la covarianza intra-clase (Intraclass Correlation). Las arquitecturas modernas crean una clase principal contenedora que encapsula ordenadamente colecciones de clases dependientes, permitiendo que las reglas lógicas matemáticas y los filtros paramétricos se apliquen de forma recursiva a la unidad experimental estratificada correspondiente.

Ecuación 3.2: Coeficiente de Correlación Intra-Clase (ICC)

\[ \text{ICC} = \frac{\sigma^2_{\text{entre}}}{\sigma^2_{\text{entre}} + \sigma^2_{\text{dentro}}} \]

donde
• \( \sigma^2_{\text{entre}} \) = varianza entre bloques y
• \( \sigma^2_{\text{dentro}} \) = varianza dentro de bloques

3.1.5 Modelos Jerárquicos en Ciencia de Datos y Políticas Públicas

Los modelos jerárquicos (también denominados modelos multinivel o multilevel models) constituyen una herramienta estadística fundamental para analizar datos agrupados o anidados en contextos de políticas públicas y ciencia de datos aplicada. Según Dhada y Labra Montes (2024), las sociedades modernas son complejas e interconectadas, y el análisis de datos desempeña un papel crítico en la comprensión de tendencias sociales. La estructura jerárquica surge naturalmente cuando las observaciones se organizan en niveles: individuos dentro de hogares, hogares dentro de vecindarios, pacientes dentro de hospitales, regiones dentro de países.

Definición 3.2 (Modelo Jerárquico Multinivel): Un modelo jerárquico aborda simultáneamente dos problemas antagónicos fundamentales de la estimación estadística:

• Alta varianza: Datos individuales insuficientes impiden predicciones confiables a nivel local.
• Alto sesgo: Agrupar (poolear) todos los datos borra la variación individual y no representa fielmente a las poblaciones reales.

La solución jerárquica modela los datos de nivel inferior con modelos individuales cuyos parámetros se derivan de distribuciones compartidas de nivel superior, logrando un equilibrio óptimo entre ambos extremos.

3.1.6 Ventajas Críticas para el Trabajo con Datos

Los modelos jerárquicos ofrecen tres ventajas metodológicas esenciales que se traducen directamente en la manera en que diseñamos y validamos estructuras de datos con Pydantic:

Ventaja Descripción Estadística Traducción en Pydantic
Estimación precisa de efectos Reconoce la estructura anidada de los datos, evitando la falacia ecológica al no extrapolar conclusiones grupales a individuos. Validadores de campo a nivel de bloque (@field_validator) y a nivel de unidad operan en sus estratos correspondientes.
Interacciones entre niveles Permite examinar cómo factores contextuales (nivel macro) modifican relaciones a nivel individual (nivel micro). Validadores del modelo contenedor pueden acceder a propiedades de los sub-modelos anidados mediante model_validator.
Préstamo de fuerza estadística Los grupos con datos escasos se benefician de la información de grupos similares (borrowing strength), mejorando la precisión predictiva. Valores por defecto (default_factory) y coerción inteligente permiten manejar datos faltantes sin rechazar bloques completos.

Caso de Estudio: COVID-19 y Modelos Jerárquicos (Dhada & Labra Montes, 2024)

Dhada y Labra Montes (2024) demuestran la potencia de los modelos jerárquicos utilizando datos públicos de COVID-19 del gobierno del Reino Unido, analizando siete regiones del NHS (National Health Service) en Inglaterra.

Problema planteado: Una de las regiones del NHS tenía datos escasos (sparse data), un desafío frecuente en enfermedades menos propagadas o en aplicaciones no médicas.

Enfoque metodológico comparativo:

  • Modelo Independiente: Cada región NHS se modela aisladamente; los coeficientes del polinomio se infieren usando únicamente los datos regionales. La región con datos escasos presenta intervalos de confianza extremadamente amplios.
  • Modelo Jerárquico: Los coeficientes polinomiales se muestrean de una distribución Gaussiana de nivel superior compartida entre regiones, reflejando el supuesto de que las regiones deberían exhibir tasas de infección similares. El resultado: "mayor precisión y confianza" mediante préstamo sistemático de información (borrowing strength).

Implicación para Políticas Públicas: Los autores concluyen que "incorporar modelos multinivel facilita el desarrollo de políticas sensibles al contexto, asegurando que las intervenciones apunten a necesidades comunitarias específicas". La implementación técnica utilizó muestreo MCMC (Monte Carlo vía Cadenas de Markov) con el motor Stan como backend, procesando los datos con Python y bibliotecas de código abierto.

3.1.8 Modelo Independiente vs. Modelo Jerárquico

La distinción fundamental entre un modelo independiente y uno jerárquico tiene implicaciones directas en cómo diseñamos nuestros esquemas de validación de datos:

Ecuación 3.3: Modelo Independiente vs. Jerárquico

Modelo Independiente: Cada grupo \( j \) se estima aisladamente:
$$ \hat{\theta}_j^{\text{indep}} = \bar{y}_j \quad \text{(estimador basado solo en datos locales)} $$

Modelo Jerárquico (Bayesiano): Las estimaciones se "encogen" hacia la media global:
$$ \hat{\theta}_j^{\text{hier}} = \lambda_j \bar{y}_j + (1 - \lambda_j) \bar{y}_{\text{global}} $$

donde \( \lambda_j \in [0, 1] \) es el factor de encogimiento (shrinkage), que depende del tamaño muestral del grupo \( j \). Con pocos datos locales, \( \lambda_j \to 0 \) y la estimación se apoya más en la distribución global.

Interpretación de \( \hat{\theta}_j^{\text{hier}} \): Es el estimador jerárquico del parámetro \( \theta \) para el grupo \( j \). A diferencia del estimador independiente \( \hat{\theta}_j^{\text{indep}} \), que utiliza únicamente los datos locales del grupo, \( \hat{\theta}_j^{\text{hier}} \) es una media ponderada entre la estimación local \( \bar{y}_j \) y la media global \( \bar{y}_{\text{global}} \). El superíndice "hier" indica que proviene de un modelo jerárquico (multinivel), en el cual los grupos no se tratan como entidades aisladas sino como miembros de una población común. Esto produce estimaciones más estables y precisas, especialmente en grupos con tamaños muestrales pequeños, gracias al principio de borrowing strength (préstamo de fuerza estadística) entre grupos.

Este principio de "préstamo de fuerza estadística" (borrowing strength) es exactamente lo que motiva el diseño de modelos Pydantic anidados: al encapsular las unidades observacionales dentro de sus bloques experimentales, preservamos la información contextual del nivel superior que permite realizar inferencias más robustas incluso cuando los datos individuales son escasos.

3.4 Caso Aplicado: Validación Jerárquica de Datos Epidemiológicos

Contexto del Problema

Inspirado en el caso de estudio de Dhada y Labra Montes (2024), donde se analizan tasas de admisión hospitalaria por COVID-19 en siete regiones del NHS en Inglaterra, el siguiente ejemplo implementa un sistema de validación jerárquica de datos epidemiológicos. La estructura de tres niveles refleja la organización natural del sistema de salud: Observaciones individuales (admisiones diarias) anidadas en Hospitales, anidados a su vez en Regiones sanitarias.

Este diseño permite que cada nivel aplique sus propias restricciones de validación, preservando la estructura jerárquica que los modelos multinivel requieren para estimar correctamente los componentes de varianza entre e intra grupos.

from typing import List, Optional
from datetime import date

# NIVEL 1 (MICRO): Observación Epidemiológica Individual
# Equivalente: Unidad observacional en diseño de parcelas divididas
# Referencia: Dhada & Labra Montes (2024) - Admisiones diarias

class VariantePatogeno(str, Enum):
    """Variantes epidemiológicas reconocidas por la OMS."""
    ALPHA = "alpha"
    DELTA = "delta"
    OMICRON = "omicron"
    ORIGINAL = "original"

class AdmisionDiaria(BaseModel):
    """
    Nivel 1 (Micro): Registro individual de admisiones hospitalarias.
    Análogo a la observación unitaria Y_ijk en el modelo jerárquico.
    """
    fecha: date
    admisiones_nuevas: int = Field(
        ..., ge=0,
        description="Número de admisiones nuevas (no puede ser negativo)"
    )
    admisiones_uci: int = Field(
        default=0, ge=0,
        description="Admisiones a UCI (subconjunto de admisiones nuevas)"
    )
    tasa_ocupacion_camas: float = Field(
        ..., ge=0.0, le=1.0,
        description="Proporción de camas ocupadas [0, 1]"
    )
    variante_dominante: VariantePatogeno = Field(
        default=VariantePatogeno.OMICRON,
        description="Variante predominante del patógeno"
    )

    @model_validator(mode='after')
    def validar_coherencia_admisiones(self):
        """
        Las admisiones UCI no pueden superar las admisiones totales.
        Restricción lógica: UCI ⊆ Admisiones Nuevas.
        """
        if self.admisiones_uci > self.admisiones_nuevas:
            raise ValueError(
                f"Incoherencia lógica: admisiones UCI ({self.admisiones_uci}) "
                f"no puede superar admisiones totales ({self.admisiones_nuevas})"
            )
        return self

# NIVEL 2 (MESO): Hospital dentro de una Región NHS
# Equivalente: Sub-parcela en diseño de parcelas divididas
# ══════════════════════════════════════════════════════════════

class HospitalNHS(BaseModel):
    """
    Nivel 2 (Meso): Hospital individual perteneciente a una región.
    Contiene series temporales de admisiones diarias.
    Análogo al efecto β_j(i) anidado en el bloque i.
    """
    codigo_hospital: str = Field(
        pattern=r'^NHS-[A-Z]{2,4}-\d{3}$',
        description="Código estandarizado: NHS-XXXX-000"
    )
    nombre_hospital: str = Field(min_length=3)
    capacidad_camas: int = Field(
        ..., gt=0, le=5000,
        description="Capacidad total de camas del hospital"
    )
    registros_diarios: List[AdmisionDiaria] = Field(
        ..., min_length=7,
        description="Mínimo 7 días de datos para estimación polinomial"
    )

    @field_validator('registros_diarios')
    @classmethod
    def validar_serie_temporal(
        cls, registros: List[AdmisionDiaria]
    ) -> List[AdmisionDiaria]:
        """
        Verifica que la serie temporal sea cronológicamente consistente.
        En modelado jerárquico (Dhada & Labra Montes, 2024),
        se usaron polinomios de tercer grado para modelar segmentos
        temporales, requiriendo datos ordenados y sin duplicados.
        """
        fechas = [r.fecha for r in registros]
        if len(set(fechas)) != len(fechas):
            raise ValueError(
                "Fechas duplicadas detectadas en la serie temporal. "
                "Cada registro debe corresponder a un día único."
            )
        if fechas != sorted(fechas):
            raise ValueError(
                "La serie temporal debe estar ordenada cronológicamente "
                "para permitir ajuste polinomial secuencial."
            )
        return registros

# NIVEL 3 (MACRO): Región Sanitaria del NHS
# Equivalente: Parcela entera (Whole-Plot) / Bloque experimental
# Referencia: 7 regiones NHS analizadas por Dhada & Labra Montes
# ══════════════════════════════════════════════════════════════

class RegionNHS(BaseModel):
    """
    Nivel 3 (Macro): Región sanitaria que contiene múltiples hospitales.
    Análogo al efecto α_i del nivel superior en Y_ijk = μ + α_i + β_j(i) + ε_k(ij)

    En el modelo jerárquico de Dhada & Labra Montes (2024), cada región
    NHS comparte parámetros de una distribución Gaussiana de nivel superior,
    implementando el concepto de 'borrowing strength'.
    """
    codigo_region: str = Field(
        pattern=r'^REG-[A-Z]{2,6}$',
        description="Código de región: REG-XXXXXX"
    )
    nombre_region: str = Field(min_length=3)
    poblacion_total: int = Field(
        ..., gt=0,
        description="Población total de la región para normalización"
    )
    hospitales: List[HospitalNHS] = Field(
        ..., min_length=2,
        description="Mínimo 2 hospitales para estimar varianza intra-región"
    )
    es_datos_escasos: bool = Field(
        default=False,
        description="Indica si la región tiene datos limitados (sparse data)"
    )

    @model_validator(mode='after')
    def validar_estructura_regional(self):
        """
        Validaciones a nivel macro que verifican la coherencia
        del bloque experimental completo.
        """
        # Verificar unicidad de códigos hospitalarios
        codigos = [h.codigo_hospital for h in self.hospitales]
        if len(set(codigos)) != len(codigos):
            raise ValueError(
                "Códigos hospitalarios duplicados en la región. "
                "Cada hospital debe tener un identificador único."
            )

        # Calcular total de observaciones para Power Analysis
        total_obs = sum(
            len(h.registros_diarios) for h in self.hospitales
        )
        if total_obs < 30:
            raise ValueError(
                f"Insuficientes observaciones regionales ({total_obs}). "
                f"Se requieren mínimo 30 para estimación robusta del ICC."
            )

        return self

    def calcular_tasa_normalizada(self) -> float:
        """
        Calcula la tasa de admisión normalizada por población,
        como en la Figura 2 de Dhada & Labra Montes (2024).
        """
        total_admisiones = sum(
            r.admisiones_nuevas
            for h in self.hospitales
            for r in h.registros_diarios
        )
        return (total_admisiones / self.poblacion_total) * 100_000

# NIVEL 4 (SUPER-MACRO): Sistema Nacional de Salud Completo
# Encapsula todas las regiones para análisis jerárquico global
# ══════════════════════════════════════════════════════════════

class SistemaNacionalSalud(BaseModel):
    """
    Nivel Superior: Contenedor de todas las regiones sanitarias.
    Permite comparaciones inter-regionales y la implementación
    del 'borrowing strength' descrito en el artículo.
    """
    pais: str = Field(default="England")
    fecha_corte: date
    regiones: List[RegionNHS] = Field(
        ..., min_length=2,
        description="Mínimo 2 regiones para modelo jerárquico"
    )

    @model_validator(mode='after')
    def validar_sistema_completo(self):
        """
        Validaciones sistémicas para garantizar que el dataset
        cumple los requisitos del modelado jerárquico multinivel.
        """
        # Verificar unicidad de regiones
        codigos_region = [r.codigo_region for r in self.regiones]
        if len(set(codigos_region)) != len(codigos_region):
            raise ValueError("Regiones duplicadas en el sistema.")

        # Alertar sobre regiones con datos escasos
        regiones_escasas = [
            r.nombre_region for r in self.regiones if r.es_datos_escasos
        ]
        if regiones_escasas:
            print(
                f"⚠ Regiones con datos escasos detectadas: "
                f"{regiones_escasas}. El modelo jerárquico aplicará "
                f"'borrowing strength' desde regiones con datos completos."
            )

        return self

Este diseño de cuatro niveles refleja directamente la estructura jerárquica analizada por Dhada y Labra Montes (2024) en su estudio de COVID-19:

Módulo 4: Manejo Integral de Errores de Validación (HTTP 422)

4.1 Fundamentos Teóricos

4.1.1 Taxonomía de Errores Computacionales en Sistemas de Datos

Comprender en profundidad la taxonomía diferencial de los errores informáticos en ecosistemas digitales es intrínsecamente análogo a auditar y clasificar los tipos de fallos de diseño metodológico durante la administración operativa de un censo nacional de población.

Definición 4.1 (Error HTTP 422 - Unprocessable Entity): Código de estado que indica que el servidor entiende el contenido de la solicitud pero no puede procesar las instrucciones contenidas. En términos estadísticos, representa una violación de las restricciones del dominio paramétrico o los axiomas del proceso generador de datos, mientras que la sintaxis de la solicitud es correcta.

4.1.2 Diferenciación entre Errores HTTP 400 y HTTP 422

Código Categoría Descripción Técnica Equivalente Metodológico
HTTP 400
(Bad Request)		 Fallo Sintáctico Estructural El motor de ingestión detecta daños lógicos severos: ausencia de comilla de cierre en JSON, fragmentación del paquete en tránsito, texto truncado antes de cargar a RAM. Ficha física de paciente destruida, incinerada o desintegrada; pérdida total de un bloque de data empírica.
HTTP 422
(Unprocessable Entity)		 Fallo Semántico de Índice El parseo crudo del JSON finaliza exitosamente, pero los validadores paramétricos detectan violaciones de dominio (ej. edad = 200 años, valor fuera de rango permitido). Sujeto que no cumple estrictamente con los criterios de inclusión protocolar; requiere rechazo formal documentado.
HTTP 409
(Conflict)		 Anomalía de Estado Causal El sistema detecta duplicidad de llave primaria, forzando retroceso (rollback) del commit. Doble conteo; el sujeto ya fue reclutado previamente y asignado a un grupo poblacional.

4.1.3 Estructura del Objeto ValidationError

Cuando Pydantic detecta una violación de validación, genera un objeto ValidationError que transporta metadatos forenses sumamente precisos:

Estructura 4.1: Campos del objeto ValidationError

ValidationError.errors() → List[Dict]
Cada diccionario contiene:
• loc: Tupla con la ruta del campo fallido (ej. ['ensayos', 2, 'cilindros'])
• msg: Mensaje descriptivo de la violación
• type: Clasificador del tipo de error (ej. 'value_error', 'type_error')
• input: Valor que causó el fallo (opcional)

4.1.4 Importancia Metodológica del Registro de Exclusiones

El registro granular de la razón por la cual un sujeto muestral fue excluido del pipeline es existencialmente crucial para cumplir con normas internacionales de publicación como las guías CONSORT o la declaración STROBE en epidemiología. Sin un registro sistemático de las exclusiones, es imposible reportar el flujo completo de participantes a través del estudio, lo cual constituye mala praxis estadística.

# 2. Clase para gestionar la bitácora de auditoría
class BitacoraAuditoria:
    """
    Sistema de registro de exclusiones para cumplimiento
    con protocolos CONSORT/STROBE.
    """
    def __init__(self):
        self.exclusiones: List[Dict[str, Any]] = []
        self.validaciones_exitosas: List[int] = []
    
    def registrar_exclusion(
        self, 
        payload_original: dict, 
        errores: list,
        timestamp: str
    ):
        """Registra una exclusión con metadatos forenses completos."""
        for error in errores:
            registro = {
                'timestamp': timestamp,
                'id_sujeto': payload_original.get('id_sujeto', 'DESCONOCIDO'),
                'campo_afectado': ' -> '.join(str(x) for x in error['loc']),
                'tipo_error': error['type'],
                'mensaje': error['msg'],
                'valor_input': error.get('input', 'N/A'),
                'payload_completo': payload_original
            }
            self.exclusiones.append(registro)
    
    def registrar_exito(self, id_sujeto: int):
        self.validaciones_exitosas.append(id_sujeto)
    
    def generar_reporte_consolidado(self) -> dict:
        """Genera resumen estadístico de las exclusiones."""
        if not self.exclusiones:
            return {'total_exclusiones': 0, 'tasa_exclusion': 0.0}
        
        # Conteo por tipo de error
        conteo_tipos = {}
        conteo_campos = {}
        for exc in self.exclusiones:
            tipo = exc['tipo_error']
            campo = exc['campo_afectado']
            conteo_tipos[tipo] = conteo_tipos.get(tipo, 0) + 1
            conteo_campos[campo] = conteo_campos.get(campo, 0) + 1
        
        total_procesados = len(self.validaciones_exitosas) + len(set(
            e['id_sujeto'] for e in self.exclusiones
        ))
        
        return {
            'total_procesados': total_procesados,
            'validaciones_exitosas': len(self.validaciones_exitosas),
            'total_exclusiones': len(self.exclusiones),
            'tasa_exclusion': len(self.exclusiones) / max(total_procesados, 1) * 100,
            'desglose_por_tipo_error': conteo_tipos,
            'desglose_por_campo': conteo_campos
        }

) -> List[ObservacionDemografica]:

    Procesa un lote de observaciones demográficas con captura

    """
    sujetos_validados = []

# Intento de validación
